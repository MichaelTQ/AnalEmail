# 163 邮箱 AI 助手

开发与维护记录请参阅 [`HANDOFF.md`](HANDOFF.md)。

## 项目用途

这是一个在 macOS 本地运行的 163 邮箱 AI 助手。目前可以只读获取和解析最近的邮件，并使用本地 Ollama/Qwen 对最新一封邮件进行结构化分析。

## 当前阶段功能

- 通过 SSL 连接 `imap.163.com:993`
- 使用邮箱地址和客户端授权码登录
- 登录后向 163 声明本地项目的 IMAP 客户端 ID，避免无标识连接被判定为 `Unsafe Login`
- 以只读方式打开 `INBOX`，按邮件时间从新到旧读取最近 10 封邮件
- 输出 UID、发件人、标题、时间、正文前 500 个字符和附件状态
- 解码常见 UTF-8、GBK、GB18030 等中英文邮件内容
- 支持 `text/plain`、`text/html` 和 multipart 邮件
- 优先使用纯文本正文；仅在没有纯文本时清理 HTML 正文
- 忽略附件内容，单封邮件解析失败时继续处理下一封
- 使用本地 Ollama/Qwen 分析最新一封成功解析的邮件
- 输出摘要、分类、重要程度、待办和截止日期，并校验 JSON 结构

程序不会删除、移动、回复、发送邮件，也不会主动设置已读标记。

## 环境要求

- macOS
- Python 3.11 或更高版本
- 已开启 IMAP 服务的 163 邮箱
- 163 邮箱客户端授权码（不是网页登录密码）
- 可访问 `imap.163.com:993` 的网络

检查 Python 版本：

```bash
python3 --version
```

输出必须为 3.11 或更高版本。当前机器的系统 `/usr/bin/python3` 是 3.9.6，但 pyenv 中已有 3.12.2；如果终端仍显示 3.9.6，可先执行：

```bash
pyenv local 3.12.2
python3 --version
```

然后再创建虚拟环境。也可以直接继续使用本项目已经由 Python 3.12.2 创建好的 `.venv`。

```bash
python -m pip install -r requirements.txt
```

## 配置 163 邮箱

1. 登录 163 邮箱网页版。
2. 进入“设置”，找到 POP3/SMTP/IMAP 等客户端服务设置。
3. 开启 IMAP/SMTP 服务。页面名称可能随 163 邮箱版本变化，请以网页版当前界面为准。
4. 按页面要求完成安全验证，并生成客户端授权码。
5. 复制授权码并妥善保存。程序登录时应使用授权码，不能使用邮箱网页登录密码。

如果账户设置中没有相关入口，先确认账号安全状态、实名或手机验证要求，并参考 163 邮箱帮助中心的最新说明。

## 创建 `.env`

复制示例文件：

```bash
cp .env.example .env
```

编辑 `.env`：

```dotenv
EMAIL_ADDRESS=你的完整163邮箱地址
EMAIL_AUTH_CODE=你生成的客户端授权码
IMAP_HOST=imap.163.com
IMAP_PORT=993
EMAIL_LIMIT=10
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=qwen3:latest
```

不要给值添加无关引号或空格，也不要把 `.env` 发给他人或提交到 Git。项目的 `.gitignore` 已排除该文件。

## 运行

确保虚拟环境已激活且当前目录为项目根目录，然后执行：

```bash
python main.py
```

正常情况下，终端会逐封显示邮件信息，并在最后显示最新邮件的 AI 分析结果：

```text
邮件解析完成：成功 10 封，失败 0 封。
最新邮件 AI 分析：
{
  "summary": "邮件摘要",
  "category": "通知",
  "importance": "中",
  "todos": [],
  "deadline": null
}
```

如果收件箱不足 10 封，则只显示实际存在的数量。

## 验收步骤

1. 确认输出包含最近最多 10 封邮件，并按时间从新到旧排列。
2. 找一封中文邮件和一封英文邮件，检查标题、发件人和正文没有明显乱码。
3. 找一封 HTML 邮件，确认预览是可读纯文本而非 HTML 标签。
4. 找一封带附件的邮件，确认“是否有附件”为“是”，正文中没有附件乱码。
5. 运行前记下一封未读邮件；运行后在网页版确认它仍是未读状态。
6. 确认末尾成功数与展示的邮件数一致；异常邮件应计入失败数且不影响后续邮件。
7. 执行 `git status --ignored`，确认 `.env` 显示为被忽略且敏感信息未出现在其他文件中。

## 常见错误排查

### 未找到 `.env`

执行 `cp .env.example .env`，然后填写真实配置。程序固定读取项目目录中的 `.env`，因此从其他目录启动也可以。

### 环境变量缺失或端口格式错误

检查 `.env` 中五项配置是否都存在。`IMAP_PORT` 和 `EMAIL_LIMIT` 必须是大于 0 的整数，端口不能超过 65535。

### 邮箱登录失败

- 确认填写的是完整邮箱地址，例如 `name@163.com`。
- 确认使用客户端授权码，而不是邮箱网页登录密码。
- 重新生成授权码后更新 `.env`，避免复制了前后空格。
- 登录网页版，确认 IMAP/SMTP 服务已开启且账号未被安全策略临时限制。

163 服务器通常不会明确区分“授权码错误”“使用了邮箱密码”和“IMAP 未开启”，所以程序会统一列出这些检查项，并且绝不会输出授权码。

### `Unsafe Login` 或登录后无法打开 `INBOX`

163 可能允许授权码登录，但拒绝没有客户端标识的后续 `EXAMINE INBOX` 操作。项目会在登录成功后、打开收件箱前发送 IMAP `ID`，如实声明项目名、版本和本地学生项目类型，不包含邮箱地址或授权码。

如果程序已经发送客户端 ID 但仍收到 `Unsafe Login`：

1. 在 163 网页端关闭 IMAP/SMTP，使旧授权码失效。
2. 重新开启服务并生成新授权码。
3. 更新 `.env` 后重试。
4. 仍然失败时，按照服务器提示联系 163 邮箱客服确认账号风控状态。

不要在聊天、代码、日志或截图中公开授权码。

### `git check-ignore` 提示不是 Git 仓库

这只表示当前项目还没有执行 `git init`，与 IMAP 连接无关。`.gitignore` 已经包含 `.env`；将来初始化 Git 后，该规则会自动生效。不要仅为测试邮箱连接而初始化或提交仓库。

### 网络连接失败

检查网络是否能访问 `imap.163.com` 的 993 端口，暂时关闭会拦截邮件协议的代理、防火墙或安全软件后重试。公共网络也可能限制 IMAP 端口。

### AI 分析失败

- 执行 `ollama list`，确认 Ollama 正在运行且存在 `.env` 中 `OLLAMA_MODEL` 指定的模型。
- 当前配置使用 `qwen3:latest`；切换模型时只需修改 `.env`，不需要改代码。
- 确认 `OLLAMA_BASE_URL=http://127.0.0.1:11434`，并且本机没有防火墙阻止该端口。
- 如果提示输出字段或 JSON 不合法，说明模型没有遵守固定格式；本阶段会拒绝该结果，不把错误数据继续传递。

### 某封邮件编码异常

程序会尝试邮件声明的字符集以及 UTF-8、GB18030、GBK、Big5 等常见编码。极少数格式损坏或使用特殊编码的邮件仍可能解析失败；程序会显示 UID 和错误类型，然后继续处理其他邮件。

### 正文为空

通知类邮件可能只有附件、远程图片或非文本 MIME 部分。程序会显示“正文为空”，这不表示连接失败。为避免附件被误读，附件和带文件名的 MIME 部分不会作为正文处理。

### 读取数量不足 10 封

确认邮件位于 `INBOX`。本阶段不会读取垃圾箱、已发送、归档或其他文件夹；收件箱不足 10 封时会返回实际数量。

## 当前阶段不支持

- 同时批量分析全部邮件
- SQLite、向量数据库或其他持久化存储
- Streamlit 或其他网页界面
- Docker、AnythingLLM
- 自动回复、自动发送、删除、移动或标记邮件
- 附件下载、附件内容解析
- `INBOX` 之外的邮箱文件夹
